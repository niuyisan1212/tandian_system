## 1. 项目概述

### 1.1 业务背景
用户每天需要探访多家店铺（探店任务）。店铺具有**时效性**（过期时间）和**探店状态**（未探店/已探店）。用户可能一天内执行多次探店行程（例如上午一批，下午一批），每次行程有独立的起点和店铺清单。系统需要：
- 集中管理店铺信息，并自动过滤已过期/已探店的店铺。
- 支持创建多个探店清单（每个清单对应一次实际出行）。
- 为每个清单自动规划最优路线，并推荐公共交通方式（公交、地铁、步行、骑行）。
- 提供可视化地图：首页展示所有待探店店铺的位置分布；选择清单后高亮该清单店铺；规划路线后在地图上绘制具体路线。
- 记录清单执行情况，完成后自动更新店铺状态。

### 1.2 技术架构
- **后端**：Java（Spring Boot 2.7+）+ MySQL 8.0 + MyBatis-Plus / JPA
- **前端**：Vue 3 + Vite + Element Plus / Ant Design Vue + 地图 SDK（高德地图 API）
- **通信**：RESTful API
- **用户认证**：无需登录，单用户私有部署

## 3. 功能性需求

### 3.1 店铺管理

| 编号 | 功能点 | 详细描述 |
|------|--------|----------|
| SHOP-01 | 添加店铺 | 表单字段：店铺名称（必填）、地址（必填）、联系电话、营业时间、备注、**过期时间**（可选，日期时间）、**探店状态**（默认“未探店”）。地址输入后，后端调用地理编码 API 自动解析经纬度并存储；解析失败时前端可手动在地图上选点。 |
| SHOP-02 | 编辑店铺 | 支持修改所有字段；若地址更改，重新触发地理编码；若手动修改坐标，标记 `geo_source = manual`。 |
| SHOP-03 | 删除店铺 | 永久删除。若该店铺已被某个未完成的清单引用，需提示用户，并自动从清单中移除（清单需重新规划）。 |
| SHOP-04 | 店铺列表 | 表格展示所有店铺，支持按“探店状态”（未探店/已探店）、“是否过期”筛选，支持按名称/地址搜索。 |
| SHOP-05 | 批量导入/导出 | 支持 Excel/CSV 导入店铺（含状态、过期时间），导出全部店铺数据。 |

**重要规则**：
- **有效待探店店铺** = `visit_status = '未探店'` AND (`expire_time` IS NULL OR `expire_time` > 当前时间)
- 路径规划、地图首页展示“所有待探店位置”时，仅包含有效待探店店铺。
- 已探店或已过期的店铺只在店铺管理页面显示，不出现在任何路线规划或地图首页的待探店标注中。

### 3.2 探店清单管理（多清单、执行与同步）

用户可以创建多个探店清单（每天数量不限），每个清单是一个独立的探店行程。

| 编号 | 功能点 | 详细描述 |
|------|--------|----------|
| LIST-01 | 创建清单 | 操作入口：清单列表页“新建清单”或地图首页快速创建。表单包含：<br> - 清单名称（如“上午场”<br> - 起点（支持地址输入、地图选点、设备定位）<br> - 从**有效待探店店铺**列表中勾选本次要探的店铺（可多选）<br> 保存后，后端自动为该清单生成路线方案（2~3种），并跳转到清单详情页。 |
| LIST-02 | 清单列表页 | 展示所有清单，按创建日期倒序分组。每个清单卡片显示：名称、日期、起点摘要、店铺数量、执行状态（待执行/已完成/已取消）。支持操作：查看详情、编辑、删除、复制清单。 |
| LIST-03 | 清单详情页 | 完整展示清单信息：<br> - 清单名称、状态（待执行/已完成/已取消）<br> - 起点地址、地图预览<br> - 店铺列表（按当前路线方案的访问顺序展示，可显示临时排除标记）<br> - 路线方案卡片（2~3张），展示总距离、总耗时、总花费，点击切换<br> - 分段交通指引（展开查看每段详细换乘说明）<br> - 可视化地图（见 3.5 节）<br> 操作按钮：<br> - **标记为已完成**：用户确认完成探店后点击，系统将所有店铺的 `visit_status` 更新为“已探店”，并记录完成时间，清单状态改为“已完成”。<br> - **标记为不执行（取消）**：放弃该行程，清单状态改为“已取消”，店铺状态不变。<br> - **重新规划**：若起点或店铺有变化（如临时排除），重新生成路线方案。<br> - **编辑清单**：修改名称、起点、选中的店铺（增减店铺后需重新规划）。 |
| LIST-04 | 多清单状态同步规则 | 当某个清单标记“已完成”时，其包含的所有店铺变为“已探店”。若其他未执行的清单中也包含这些店铺，系统需在那些清单的详情页给出提示（如“店铺 XXX 已被其他清单完成，请重新规划”），并建议用户重新规划（自动过滤已探店店铺）。 |
| LIST-05 | 临时排除店铺 | 在清单详情页的店铺列表中，用户可勾选“临时排除”个别店铺（不参与本次路线规划）。排除后，系统需重新规划路线（用户需手动点击“重新规划”按钮）。 |
| LIST-06 | 清单日历视图（可选） | 提供月历/周历视图，直观展示每天有哪些清单及其状态（待执行/已完成/已取消）。 |

### 3.3 路线规划与多方案生成

| 编号 | 功能点 | 详细描述 |
|------|--------|----------|
| ROUTE-01 | 规划触发 | 创建清单时自动规划；在清单详情页点击“重新规划”时重新计算。 |
| ROUTE-02 | 方案数量 | 生成 2~3 条不同策略的路线：<br> - **时间优先**：总耗时最短，优先地铁/快速公交<br> - **成本优先**：总花费最低，优先普通公交<br> - **均衡方案**：时间与成本折中 |
| ROUTE-03 | 路线详情 | 每条方案展示：<br> - 总距离（公里）<br> - 预计总耗时（分钟，含步行、等车、乘车）<br> - 预估总花费（元，公交/地铁按城市标准计费，步行/骑行免费）<br> - 店铺访问顺序（起点 → 店铺A → 店铺B → … → 最后一个店铺） |
| ROUTE-04 | 分段交通指引 | 对每段行程（起点→第一店、店→店）给出：<br> - 交通方式（步行/骑行/公交/地铁）<br> - 具体路线：如“步行 300 米至人民广场站，乘 1 号线（莘庄方向）2 站至黄陂南路站，1 号口出步行 200 米”<br> - 预估耗时与距离 |
| ROUTE-05 | 方案切换 | 用户在清单详情页点击不同方案卡片时，地图上立即切换绘制对应的路线，店铺编号按新顺序刷新。 |

### 3.4 公共交通推荐细节

| 编号 | 功能点 | 详细要求 |
|------|--------|----------|
| TRANS-01 | 允许的交通方式 | 仅限 **步行、骑行、公交、地铁**。不得推荐打车、自驾、轮渡等其他方式。 |
| TRANS-02 | 换乘信息 | 每段公交/地铁行程需包含：线路名称、上车站点、下车站点、乘坐站数/时间、换乘步行距离（若有多段换乘则逐一列出）。 |
| TRANS-03 | 备选方案 | 对每段行程，除主推荐外，提供一个备选交通方式（例如主推荐地铁，备选推荐公交）。 |
| TRANS-04 | 骑行决策 | 当两点间直线距离或路网距离在 0.8km ~ 3km 之间且道路允许骑行时，优先推荐骑行；否则使用步行、公交或地铁。 |

地图模块分为两个核心场景：**地图首页**（默认首页）和 **清单详情页**。

#### 3.5.1 地图首页（全局视图）

| 编号 | 功能点 | 详细描述 |
|------|--------|----------|
| MAP-HOME-01 | 展示所有待探店位置 | 地图自动加载并标注所有 **有效待探店店铺**（按 3.1 规则）。统一使用圆形或店铺图标，点击弹出信息窗（名称、地址、状态）。 |
| MAP-HOME-02 | 清单选择器 | 在页面顶部或侧边栏放置下拉选择器，展示所有已创建的探店清单（按日期倒序）。用户可选择任一清单。 |
| MAP-HOME-03 | 清单高亮与置灰 | 当用户选择一个清单后：<br> - 地图上 **高亮** 该清单中包含的店铺（图标放大、加彩色外圈或亮度提高）<br> - **其他所有店铺置灰**（半透明或灰色图标）<br> - 若该清单有起点，同时显示起点图标（⭐）<br> - 未选择清单时，所有有效店铺正常显示。 |
| MAP-HOME-04 | 自适应视野 | 选择清单后，地图自动缩放至包含该清单所有店铺及起点的最佳视野。 |
| MAP-HOME-05 | 快速创建清单 | 在地图首页，用户可多选店铺（Ctrl+点击或勾选模式），然后点击“创建清单”按钮，跳转到创建表单并预填所选店铺。 |

#### 3.5.2 清单详情页（路线视图）

| 编号 | 功能点 | 详细描述 |

| MAP-DETAIL-02 | 路线绘制 | 根据当前选中的方案（时间优先/成本优先/均衡），绘制完整路线。不同交通方式用不同线型颜色：<br> - 步行：绿色实线<br> - 骑行：橙色虚线<br> - 公交：蓝色实线<br> - 地铁：红色实线 |

## 5. 店铺经纬度存储方案

**设计目标**：避免每次路线规划都调用地理编码 API，提升性能并降低配额消耗。

**店铺表相关字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `longitude` | DECIMAL(10,7) | 经度（GCJ-02），必填 |
| `latitude` | DECIMAL(10,7) | 纬度（GCJ-02），必填 |
| `geo_source` | VARCHAR(20) | 坐标来源枚举：`geocode`（地址解析）、`manual`（地图选点）、`accurate`（精确测绘） |

**交互流程**：
1. **添加店铺**：用户输入地址 → 后端调用地图地理编码 API → 若成功，存储经纬度及 `geo_source='geocode'`；若失败（地址模糊），前端提示用户手动在地图上选点，选点后存储坐标及 `geo_source='manual'`。
2. **编辑店铺**：若地址被修改，重新触发地理编码并更新经纬度；若用户手动移动地图标注点，更新坐标并标记 `geo_source='manual'`。
3. **验证**：存储前确保经纬度不为空（地址解析失败且用户未手动选点时禁止保存）。

**优势**：
- 规划路线时直接使用已存坐标，无需实时解析地址。
- 支持离线环境（如果地图 API 不可用，至少已有坐标可进行 TSP 计算和距离矩阵调用，但距离矩阵仍需 API）。

## 6. 数据模型设计

### 6.1 店铺表 `shop`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| name | VARCHAR(100) | NOT NULL |
| address | VARCHAR(255) | 地址文本 |
| longitude | DECIMAL(10,7) | NOT NULL |
| latitude | DECIMAL(10,7) | NOT NULL |
| geo_source | VARCHAR(20) | 默认 'geocode' |
| phone | VARCHAR(20) | 可选 |
| business_hours | VARCHAR(100) | 可选 |
| remark | TEXT | 可选 |
| visit_status | TINYINT | 0=未探店, 1=已探店，默认0 |
| expire_time | DATETIME | NULL 表示永不过期 |
| created_at | DATETIME | 默认 CURRENT_TIMESTAMP |
| updated_at | DATETIME | 默认 CURRENT_TIMESTAMP ON UPDATE |

### 6.2 探店清单表 `shopping_list`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| name | VARCHAR(100) | 清单名称 |
| task_date | DATE | 清单所属日期（用户创建时默认当天） |
| start_lng | DECIMAL(10,7) | 起点经度 |
| start_lat | DECIMAL(10,7) | 起点纬度 |
| start_address | VARCHAR(255) | 起点地址文本 |
| status | TINYINT | 0=待执行, 1=已完成, 2=已取消 |
| selected_plan_id | VARCHAR(50) | 用户最终选择的方案标识，如 'time_first' |

## 8. 用户界面设计要点

### 8.1 页面结构

| 页面 | 路由 | 说明 |
|------|------|------|
| 地图首页 | `/` | 默认首页，展示所有待探店店铺 + 清单选择器 + 高亮功能 |
| 清单列表页 | `/lists` | 展示所有探店清单，支持增删改查、筛选 |
| 清单详情页 | `/lists/:id` | 展示单个清单的完整信息，包含路线方案、地图、操作按钮 |
| 店铺管理页 | `/shops` | CRUD 店铺，表格展示，批量导入导出 |

### 8.2 地图首页具体布局

- **顶部栏**：标题“探店地图”、清单选择下拉框（宽度适中）、新建清单按钮。
- **侧边栏（可选，可折叠）**：显示当前选中清单的简要信息（名称、状态、店铺列表预览）。移动端改为底部抽屉。
- **地图主体**：占满剩余空间，展示标注和路线（若有选中清单且有规划结果，则显示路线；否则只显示店铺标注）。
- **右下角悬浮按钮**：定位到当前位置（调用浏览器定位）。

### 8.3 清单详情页布局

- **顶部**：清单名称、状态标签、操作按钮（标记完成/取消/编辑/重新规划）。
- **左侧/上方（PC为左侧，移动为顶部折叠区）**：起点信息卡片、店铺列表（可排序、临时排除）。
- **右侧/下方（PC为右侧，移动为下方）**：地图（占较大区域）。
- **底部**：方案卡片横向滚动（2~3张），点击切换；展开详细视图显示分段交通指引。

### 8.4 关键交互流程

**流程一：从地图首页快速创建清单**
1. 用户在地图首页点击“框选”或“多选”模式（可用 Ctrl+单击多选）。
2. 选择多个店铺后，点击“创建清单”按钮。
3. 弹出侧边表单，自动填充所选店铺，用户填写清单名称、起点。
4. 保存后，后端规划路线，跳转到清单详情页。

**流程二：在清单详情页完成探店**
1. 用户实际执行行程后，返回清单详情页，点击“标记为已完成”。
2. 弹窗二次确认：“将标记该清单内所有店铺为已探店，是否继续？”
3. 确认后，后端更新所有店铺状态为已探店，清单状态改为已完成，页面变为只读模式（不可再规划或编辑）。

**流程三：清单选择器联动地图首页**
1. 用户在地图首页下拉框选择一个清单。
2. 地图高亮该清单的店铺（其他置灰），缩放至合适视野。
3. 如果该清单已有路线方案（之前规划过），地图上绘制路线；否则只显示高亮店铺。
4. 用户可点击清单旁“查看详情”跳转至详情页。

## 9. 算法逻辑简述

### 9.1 有效店铺筛选
- 从 `shop` 表中选出 `visit_status = 0` 且 (`expire_time` IS NULL OR `expire_time` > NOW()) 的店铺。

### 9.2 距离矩阵计算
- 针对起点和所有选中店铺，两两之间调用**步行路径规划 API**（因为探店最终依赖步行接驳，步行距离最能反映实际代价）。
- 返回步行距离（米）和步行时间（秒），构建代价矩阵。
- 注意：起点到店铺、店铺到店铺均使用步行 API。

### 9.3 路径顺序优化（TSP）
- 采用**最近邻算法**（起始点为起点）生成初始路径，然后使用 **2-opt** 局部优化减少总步行距离。
- 时间复杂度 O(n²)，适用于 n ≤ 20。
- 输出店铺访问顺序列表（不包含回到起点）。

### 9.4 多策略方案生成

| 方案类型 | 生成方法 |
|----------|----------|
| **均衡方案** | 使用上述 TSP 顺序，然后对每段行程根据距离阈值选择交通方式（见 9.5）。 |
| **时间优先** | 修改代价矩阵：将步行时间替换为**公共交通预估时间**（调用公交/地铁 API 的 duration）。重新运行 TSP 得到顺序，再按时间优先的交通模式（长距离强制地铁）生成分段。 |
| **成本优先** | 使用 TSP 顺序（基于步行距离），但每段行程强制优先使用普通公交（成本最低），仅当公交不可达时使用地铁或步行/骑行。 |

### 9.5 分段交通决策规则

对于一段行程（距离 `d` 米）：

| 条件 | 推荐方式 | 说明 |
|------|----------|------|
| `d ≤ 800` | 步行 | 直接调用步行 API 获取路线 |
| `800 < d ≤ 3000` | 骑行（优先） | 调用骑行 API；若骑行不可达（如快速路），则降级为公交 |
| `d > 3000` | 公共交通（公交/地铁） | 调用公共交通 API，取 API 返回的第一个方案；时间优先时偏好地铁，成本优先时偏好公交 |

- 对于公共交通方案，需解析 API 返回的换乘步骤，提取线路、站点、步行距离等，生成 `instruction` 文本。
- 备选交通方式：取公共交通 API 返回的第二方案（如果存在），或者骑行/步行的组合。

## 10. 约束与假设

- **地理范围**：中国大陆城市，坐标系统一为 **GCJ-02**。
- **店铺数量**：单个清单最多支持 20 个店铺（超过时前端给出性能警告，但可继续）。
- **地图 API 配额**：假设使用高德地图个人开发者 Key，每日总调用量充足（通常 10 万次/日）。建议对常用路线结果做缓存（Redis 或内存）。
- **实时性**：不要求实时公交到站信息，公共交通方案基于静态时刻表/推荐路线。
- **数据备份**：用户需自行备份数据库（系统可提供导出功能）。

## 11. 未来优化方向

以下功能已在需求中预留数据或逻辑扩展点，后续可按需开发：

1. **智能出发时间推荐**：基于历史清单的实际耗时和时段拥堵数据，推荐最佳出发时间。
2. **探店报告自动生成**：根据已完成清单，生成图文报告（探店效率、里程分布、花费统计），支持导出 PDF。
3. **主动学习用户偏好**：收集用户每次选择的方案类型（时间优先/成本优先/均衡）以及手动调整顺序的行为，动态调整 TSP 代价权重（如增加换乘惩罚系数、偏好地铁等）。
4. **店铺标签与分类**：为店铺添加自定义标签（如“需要预约”“有停车位”），影响路线权重（例如需预约的店铺安排在特定时段）。
5. **多用户支持**：增加登录体系、角色权限，将数据隔离到不同用户。
6. **实时公交集成**：对接实时公交 API，在规划时优先推荐到站时间匹配的线路。
